Android笔记：Activity组件

发表于 2017-10-18 | 分类于 技术

Activity 是一个用来显示和与用户交互的组件。既可以全屏显示，也可小于屏幕并浮动在其他窗口之上。
本篇对 Activity 相关知识点的个人总结，了然于心，才能得心应手。

基本状态

触发不同的生命周期会导致不同的状态，有三种状态：

  1. 运行中（优先级1）： Activity 可见并具有用户焦点。

  2. 暂停：（优先级2） 可见或部分可见状态。 暂停的 Activity 处于完全活动状态（Activity 对象保留在内存中，它保留了所有状态和成员信息，并与窗口管理器保持连接），但在内存极度不足的情况下，可能会被系统终止。

  3. 停止（优先级3）： 不可见状态。已停止的 Activity 同样仍处于活动状态（Activity 对象保留在内存中，它保留了所有状态和成员信息，但未与窗口管理器连接）。 对用户不再可见，在他处需要内存时可能会被系统终止。

注：系统内存资源不足时，会按优先级由低到高销毁

生命周期

点击查看周期图

onCreate

首次创建 Activity 时调用。用来做一些初始化操作。始终后接 onStart()。

onRestart

在 Activity 已停止并即将再次启动前调用。后接 onStart()

onStart

在 Activity 显示之前调用。如 Activity 转入前台，则后接 onResume()，如转入隐藏状态，则后接 onStop()。

onResume

在 Activity 已显示并在交互之前调用。具有输入焦点。始终后接 onPause()。

onPause

即将开始继续另一个 Activity 时调用。应该非常迅速地执行所需操作，因为它返回后，下一个 Activity 才能继续执行。如果 Activity 返回前台，则后接 onResume()，如果 Activity 转入对用户不可见状态，则后接 onStop()。

onStop

对用户不再可见时调用。如果被销毁，或另一个 Activity 继续执行并将其覆盖，就可能发生这种情况。如果恢复与用户的交互，则后接 onRestart()，如果被销毁，则后接 onDestroy()。

onDestroy

在被销毁前调用。当 Activity 结束（对 Activity 调用了 finish()），或系统为节省空间而暂时销毁该 Activity 实例时，可能会调用它。 可以通过 isFinishing() 方法区分这两种情形。

其他周期

一个完整的生命周期，除了上边常用的周期回调，也包括如下几个回调接口

  • onContentChanged() 当Activity的布局改动时
  • onPostCreate() onCreate 方法彻底执行完毕的回调
  • onPostResume() onResume 方法彻底执行完毕的回调
  • onConfigurationChanged() 在配置文件 AndroidManifest 中处理了 configChanges属性对应的设备配置，该方法才会被调用
  • onSaveInstanceState() 保存状态信息
  • onRestoreInstanceState() 恢复状态信息

执行顺序：onCreate >>> onContentChanged >>> onStart >>> onPostCreate >>> onResume >>> onPostResume >>> onPause >>> onStop >>> onDestroy

保存状态

在后台运行多个任务，并系统内存资源不足时，会销毁一些低优先级（优先级对应上述3种状态）的 Activity。当用户返回时会重建这个Activity,用户并不知道这个过程。因为重建了Activity,因此一些状态会信息会丢失。 在这种情况下，实现onSaveInstanceState()对状态的信息进行保存。

系统在销毁 Activity 前，会先调用 onSaveInstanceState()，系统会向该方法传递一个 Bundle，调用它的 putString() 和 putInt() 等方法以保存有关 Activity状态的信息。然后如系统终止应用进程，并且用户返回 Activity，则系统会重建该 Activity，并将 Bundle 同时传递给 onCreate() 和 onRestoreInstanceState()。从 Bundle 提取保存的状态并恢复。如果没有状态信息需要恢复，则传递 Bundle 是空值（首次创建该 Activity 这种情况）。

如用户显式关闭 Activity ，系统在销毁前则不会调用 onSaveInstanceState(),如系统调用 onSaveInstanceState 的情况，是在 onStop() 也可能在 onPause() 之前调用。

Activity 默认的 onSaveInstanceState() 实现对依次调用布局中View 的 onSaveInstanceState()。每个View 需要有有提供id属性，系统在重建ui时才会恢复其状态。

启动模式

Android 提供了四种 Activity 启动方式：

  1. 标准模式（standard）
    默认。系统在启动 Activity 的任务中创建 Activity 的新实例并向其传送 Intent。Activity 可以多次实例化，而每个实例均可属于不同的任务，并且一个任务可以拥有多个实例。

  2. 栈顶复用模式（singleTop）
    当前任务栈的顶部已存在 Activity 的一个实例，则系统会通过调用该实例的 onNewIntent() 方法向其传送 Intent，不会创建新的实例。也不会调用其他生命周期的方法。如不在栈顶，则创建一个新实例，放入栈顶。

  3. 栈内复用模式（singleTask）
    该模式是一种单例模式，即一个栈内只有一个实例。和 singleTop 的区别是，不管是不是在栈顶，只要栈内有实例就会复用。singleTask具有clearTop特性，把之上的栈内Activity清除。

  4. 单例模式（singleInstance）
    与 singleTask 相同，只是启动的任何 Activity 均在单独的任务中打开，并该 Activity 是其任务唯一仅有的成员

常用 Flags

可以在启动 Activity 时，通过 Intent 的 addFlags() 方法设置 Activity 的启动模式。

FLAG_ACTIVITY_NEW_TASK
其效果与指定 Activity 为 singleTask 模式一致。

FLAG_ACTIVITY_SINGLE_TOP
其效果与指定 Activity 为 singleTop 模式一致。

FLAG_ACTIVITY_CLEAR_TOP
销毁目标Activity和它之上的所有 Activity，重新创建目标 Activity

任务栈

Activity 的管理是采用任务栈的形式，任务栈采用“后进先出”的栈结构。如：
当 Activity 启动另一个 Activity 时，该新 Activity 会被推送到堆栈顶部，成为焦点所在。 前一个 Activity 仍保留在堆栈中，但是处于停止状态。Activity 停止时，系统会保持其用户界面的当前状态。 用户按“返回”按钮时，当前 Activity 会从堆栈顶部弹出并销毁，而前一个 Activity 恢复执行。
可以使用下列几个 Activity 属性修改此行为：

清理任务栈

如果用户长时间离开任务，则系统会清除所有 Activity 的任务，根 Activity 除外。 当用户再次返回到任务时，仅恢复根Activity。系统这样做的原因是，经过很长一段时间后，用户可能已经放弃之前执行的操作，返回到任务是要开始执行新的操作。

  1. alwaysRetainTaskState
    如果在任务的根 Activity 中将此属性设置为 “true”，即使在很长一段时间后，任务仍将所有 Activity 保留在其堆栈中。

  2. clearTaskOnLaunch
    如果在任务的根 Activity 中将此属性设置为 “true”，则每当用户离开任务然后返回时，系统都会将堆栈清除到只剩下根 Activity。它与 alwaysRetainTaskState 正好相反。 即使只离开任务片刻时间，用户也始终会返回到任务的初始状态。

  3. finishOnTaskLaunch
    此属性类似于 clearTaskOnLaunch，但它对单个 Activity 起作用，而非整个任务。如设置为 true 用户一旦离开了当前任务，再次返回时这个Activity就会被清除掉。

TaskAffinity 关联处理

“关联”指 Activity 优先属于哪个任务。 默认情况下，Activity 没有指明 taskAffinity，那么它就等于 Application 指明的taskAffinity，如 Application 也没有指明，那么该 taskAffinity 的值就等于应用的包名。这个属性的值是一个字符串，可以指定为任意字符串，但是必须至少包含一个”.”。在不同应用中定义的 Activity 可以共享关联，或者可为在同一应用中定义 Activity 分配不同的任务关联。TaskAffinity 属性主要和singleTask 启动模式或者 allowTaskReparenting 配合使用，才会起作用。这两种情况分别如下：

  1. 如果加载某个 Activity 的 intent，Flag 被设置成 FLAG_ACTIVITY_NEW_TASK时，它会首先检查是否存在与自己 taskAffinity 相同的 Task，如果存在，那么它会直接宿主到该Task中，如果不存在则重新创建 Task。

  2. 如果该 Activity 的 allowTaskReparenting 设置为 true，它进入后台，当一个和它有相同 affinity 的 Task 进入前台时，它会重新宿主，进入到该前台的 task 中。

使用总结

  1. 合理的利用各个周期回调，安排相应功能的代码，不在周期函数内做有阻塞或耗时的操作。避免引起界面卡顿和无响应anr的情况。
  2. Activity 除了被主动关闭外，会后各种异常情况，重建销毁。根据页面情况，合理的实现 onSaveInstanceState() 和 onRestoreInstanceState() 接口。
  3. 如有屏幕旋转或其他的一些系统设置改变，会触发 onConfigurationChanged(),如不想重复进入 onCreate() ,则需要在 Android:configChanges 属性中设置需要捕获的事件类型
  4. 如需传递大量对象，最好封装成对象的方式传递。保证接口整洁
  5. 如一个公共的页面，可能会被很多个地方触发启动，并接收不同的参数，最好统一在被启动的 Activity 中提供静态方法启动入口。好对调用参数有个统一管理，也好区分有几种参数的接收形式，方便后期维护。
  6. 非 Activity 类型 Context (如 Service) 中启动 Activity。需要指定FLAG_ACTIVITY_NEW_TASK标记。（因为其它 Context 并不会有任务栈，所以要创建一个新任务栈启动）
