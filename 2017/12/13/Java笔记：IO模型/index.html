Java笔记：IO模型

发表于 2017-12-13 | 分类于 技术

一般分为内存IO,网络IO,和磁盘IO。其中内存IO的速度远快于后两者，性能瓶颈一般不在于内存IO.由于IO操作都是由统调用通过cpu来调度的，而cpu的速度远快于IO操作，导致浪费cpu的宝贵时间来等待慢速的IO操作。为了让cpu和慢速的IO设备更好的协调工作，减少CPU在IO调用上的消耗，逐渐发展出各种IO模型。

IO的过程

一般分为两个阶段，应用程序向操作系统发出IO请求给操作系统内核，操作系统内核需要等待数据就绪，这里的数据可能来自别的应用程序或者网络
等待数据：数据可能来自其他应用程序或者网络，如果没有数据，操作系统就一直等待，应用程序就跟着等待。
拷贝数据：将就绪的数据读取进内核态内存，在拷贝到用户态内存。

五种IO模型

JAVA作为运行在宿主机上的程序，底层也遵循这5中I/O模型规则。

  1. 阻塞式IO

    程序发起IO操作请求后 ，如数据没有准备就绪，就一直等待，直到数据准备就绪。

  2. 非阻塞式IO

    程序发起IO操作请求后，不需要等待，立即返回一个标志信息。等到数据准备就绪以后，内核会把数据拷贝到线程中去。需要不断询问内核是否已经准备好数据，轮询也会消耗大量CPU时间，因此这种模式并不常用。

  3. 多路复用

    非阻塞IO将会轮询一个IO是否可用，而多路复用则是轮询多个IO是否至少有一个可用。多路复用IO实现主要包括四种：select（Window/Linux）、poll(Linux)、epoll(Linux)、kqueue(Linux)。

  4. 信号驱动式IO

    程序发起IO请求时，可以给IO请求注册一个信号函数，请求立即返回，操作系统底层则处于等待状态（等待数据就绪），直到数据就绪，然后通过信号通知主调程序。信号驱动也是一种非阻塞式的IO模型，比起上面的非阻塞式IO模型，信号驱动式IO模型不需要轮询检查底层IO数据是否就绪，而是被动接收信号，比起多路复用IO模型来说，信号驱动IO模型针对的是一个IO的完成过程， 而多路复用IO模型针对的是多个IO同时进行时候的场景。

  5. 异步IO

    异步IO模型是将整个IO操作（包括等待数据就绪，复制数据到应用程序工作空间）全都交给操作系统完成，操作系统完成整个过程之后，再通知应用程序。
    异步IO模型跟信号驱动IO模型很相似，区别是信号驱动是在数据就绪的时候通知应用程序，而异步IO模型则是数据就绪且操作系统已经将数据拷贝进应用程序运行空间之后，系统再通知应用程序，这个过程中应用程序不需要阻塞。
    相比于多路复用模型，异步IO并不十分常用，不少高性能并发服务程序使用IO多路复用模型+多线程任务处理的架构基本可以满足需求。目前操作系统对异步IO的支持并非特别完善，更多的是采用IO多路复用模型模拟异步IO的方式（IO事件触发时不直接通知用户线程，而是将数据读写完毕后放到用户指定的缓冲区中）。Java7之后已经支持了异步IO

同步和异步

前面四种IO模型中，应用程序都会在某一环节阻塞（即使是轮询，也算是一种阻塞），POSIX将这种IO模型称为同步IO操作，而异步IO模型，则是全权把IO操作整个过程都交给操作系统，中途无阻塞，POSIX将这种IO模型称为异步IO操作。
Windows系统提供了一种异步IO技术：IOCP（I/O Completion Port，I/O完成端口,Linux下由于没有这种异步IO技术，所以使用的是epoll模拟。

JAVA中的IO

  • JAVA 1.4版本中加入的 JAVA NIO 对多路复用IO的实现
  • JAVA 1.7版本中加入的NIO2.0（AIO）对异步IO的实现

处理器类 EchoServerHandler.java

问题：每次接收到新的连接都要新建一个线程，处理完成后销毁线程，代价大。当有大量地短连接出现时，性能比较低

改进为“阻塞I/O+线程池”模式

主线程 ThreadPoolEchoServer.java

问题：在大量短连接的场景中性能会有提升，因为不用每次都创建和销毁线程，而是重用连接池中的线程。但在大量长连接的场景中，因为线程被连接长期占用，不需要频繁地创建和销毁线程，因而没有什么优势。
这种方法可以适用于小到中度规模的客户端的并发数，如果连接数超过 100,000或更多，那么性能将很不理想

####
改进为“非阻塞I/O”模式
“阻塞I/O+线程池”网络模型虽然比”阻塞I/O+多线程”网络模型在性能方面有提升，但这两种模型都存在一个共同的问题：读和写操作都是同步阻塞的,面对大并发（持续大量连接同时请求）的场景，需要消耗大量的线程来维持连接。CPU 在大量的线程之间频繁切换，性能损耗很大。一旦单机的连接超过1万，甚至达到几万的时候，服务器的性能会急剧下降。

而 NIO 的 Selector 却很好地解决了这个问题，用主线程（一个线程或者是 CPU 个数的线程）保持住所有的连接，管理和读取客户端连接的数据，将读取的数据交给后面的线程池处理，线程池处理完业务逻辑后，将结果交给主线程发送响应给客户端，少量的线程就可以处理大量连接的请求。

Java NIO 由以下几个核心部分组成：

  • Channel
  • Buffer
  • Selector
    要使用 Selector，得向 Selector 注册 Channel，然后调用它的 select()方法。这个方法会一直阻塞到某个注册的通道有事件就绪。一旦这个方法返回，线程就可以处理这些事件，事件的例子有如新连接进来，数据接收等。

使用总结

BIO方式适用于连接数量小，连接时间短，计算密集，代码编写直观，程序直观简单易理解，JDK1.4之前。
NIO方式适用于连接数量大，连接时间短，比如Http服务器，并发局限于应用中，编程比较复杂，JDK1.4开始支持。
AIO方式使用于连接数量大，连接时间长，IO密集型，比如聊天服务器，充分调用OS参与并发操作，编程比较复杂，JDK7开始支持。

第三方框架

虽然JAVA NIO 和 JAVA AIO框架提供了 多路复用IO/异步IO的支持。要编写一个可靠的、易维护的、高性能的 NIO/AIO 服务器应用 ,使用 Netty 和 MINA 等框架会更方便稳定。框架本身兼容实现各类操作系统的实现，修复了Java NIO 框架存在一个poll/epoll BUG。并对上层支持了更多特有的服务方法。
